## 첫 cleanup PR 후보

1. PR 1: terminal status/domain outcome rename inventory 적용
   - `cache_hit` terminal status를 `terminalStatus=success + cache.outcome=hit + provider.outcome=not_called`로 매핑한다.
   - `error`를 v2 `failed`로 옮기고, auth failure는 `blocked + httpStatus/errorCode`로 분리하는 migration plan을 만든다.

2. PR 2: safety outcome/summary mapping 문서화
   - `safety.outcome`을 canonical 값으로 두고 `maskingAction`, `detectedTypes`, `detectedCount`, `redactedPromptPreview`의 위치와 RBAC를 확정한다.
   - `credential` 같은 demo category를 detector category summary로 둘지, 계약 detector type으로 정규화할지 결정한다.

3. PR 3: raw-derived hash visibility cleanup
   - `promptHash`, `requestBodyHash`, `cacheKeyHash`의 API/UI/metrics 노출 범위를 분리한다.
   - Semantic Cache evidence 지표를 actual Exact Cache hit/saved cost/provider bypass 지표와 분리한다.
