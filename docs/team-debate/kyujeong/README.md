# Kyujeong

이 디렉토리는 Kyujeong의 GateLM 설계/기술 토론 메모를 정리하는 공간입니다.

## 토론 전제

남은 시간이 아직 4일 정도 있으므로, 지금은 기능을 미리 포기하거나 범위를 닫아버릴 단계가 아니라고 봅니다.

대신 현재 구현된 기반을 기준으로, 앞으로 어디에 시간을 쓰면 v1.0.0의 설득력이 가장 커지는지 토론하고 싶습니다.

핵심 질문은 이것입니다.

> 지금 이미 만들어진 Gateway 기반 위에, 남은 시간 동안 무엇을 더 붙이면 "이건 실제로 쓸 수 있는 LLM 운영 Gateway다"라는 인상을 줄 수 있을까?

## 현재 구현된 것으로 보이는 기반

현재 repository를 기준으로 보면 GateLM은 이미 단순 아이디어 단계는 넘어섰습니다.

- Gateway 서버 기본 구조가 있다.
- health / readiness 확인 흐름이 있다.
- OpenAI-compatible Chat Completions 흐름이 있다.
- 모델 목록을 반환하는 흐름이 있다.
- API Key와 App Token 기반 인증 흐름이 있다.
- Tenant / Project / Application 맥락을 요청에 연결하는 구조가 있다.
- 민감정보를 Provider 호출 전에 가리거나 차단하는 흐름이 있다.
- 동일한 안전 요청에 대해 Exact Cache를 적용하는 흐름이 있다.
- `model=auto` 요청에 대해 단순 모델 선택을 수행하는 흐름이 있다.
- Mock Provider를 통한 end-to-end 호출 기반이 있다.
- 요청 성공, 차단, 인증 실패, 캐시 결과를 로그로 남기는 기반이 있다.
- Request Log 목록, Request Detail, Dashboard Overview를 조회하는 기반이 있다.
- PostgreSQL / Redis / Mock Provider 기반 로컬 실행 환경이 있다.
- smoke script와 테스트 파일들이 꽤 많이 쌓여 있다.

즉, 지금의 토론은 "처음부터 뭘 만들까"가 아니라 "이미 있는 Gateway 흐름을 어떻게 제품처럼 보이게 만들까"에 가까워야 합니다.

## 지금부터 토론하고 싶은 방향

### 1. v1.0.0의 메시지

GateLM을 한 문장으로 설명할 때 무엇을 앞세울지 정해야 합니다.

후보:

- 기업 LLM 요청을 한 곳에서 인증, 보호, 추적하는 Gateway
- 민감정보 유출을 막고 LLM 비용을 줄이는 운영 레이어
- Provider를 직접 쓰는 대신, 조직 정책을 통과한 안전한 LLM 호출 경로
- 개발팀이 기존 LLM 호출을 크게 바꾸지 않고 운영 통제를 붙이는 방법

토론 질문:

- 비용 절감, 보안, 관찰 가능성 중 무엇을 첫 메시지로 잡을까?
- "LLMOps"라는 큰 말보다 "Gateway"라는 구체적 메시지가 더 강하지 않을까?
- 고객 입장에서 가장 먼저 체감할 가치는 비용일까, 보안일까, 로그일까?

### 2. 남은 4일 동안 가장 임팩트 있는 추가 작업

지금 기반이 이미 Gateway 흐름 중심으로 잡혀 있으니, 남은 시간은 제품 인상을 키우는 데 써야 한다고 봅니다.

붙여볼 만한 후보:

- 고객사 앱 연동 데모를 더 실제 사용처럼 다듬기
- Web Console 또는 간단한 운영 화면을 정리하기
- 실제 Provider 1개를 연결해서 Mock Provider와 비교 시연하기
- Dashboard Overview를 더 설득력 있는 운영 지표로 다듬기
- Request Detail을 운영자가 납득할 수 있게 정리하기
- Rate Limit 또는 Budget 관련 최소 시나리오를 붙이기
- 에러/timeout/fallback 상황을 데모에 포함하기
- 데모 시나리오를 발표용으로 매끄럽게 만들기

토론 질문:

- 남은 시간에 "보이는 제품성"을 키우는 게 좋을까, "Gateway 내부 완성도"를 높이는 게 좋을까?
- 실제 Provider 연동은 데모 신뢰를 크게 올릴까, 아니면 장애 변수만 늘릴까?
- Rate Limit / Budget은 지금 붙이면 제품 메시지가 강해질까?
- 운영 화면을 강화하는 것이 백엔드 기능 추가보다 더 큰 설득력을 만들 수 있을까?

### 3. 현재 구현 기반에서 더 강하게 보여줄 수 있는 것

이미 있는 기능도 보여주는 방식에 따라 완성도가 달라질 수 있습니다.

강하게 보여줄 수 있는 장면:

- 정상 요청이 Gateway를 통과해 Provider 응답으로 돌아오는 장면
- 잘못된 인증 정보가 Provider 호출 전에 막히는 장면
- 민감정보가 포함된 요청이 redaction 또는 block 되는 장면
- 같은 요청의 두 번째 호출에서 캐시 효과가 나는 장면
- `model=auto`가 실제 선택 모델로 바뀌고, 그 이유가 남는 장면
- 요청 로그에서 성공, 차단, 캐시 결과가 구분되는 장면
- Request Detail에서 원문 없이도 운영 판단이 가능한 장면
- Dashboard에서 전체 요청, 성공, 차단, 캐시 효과가 한눈에 보이는 장면

토론 질문:

- 이 중 발표에서 가장 먼저 보여줘야 할 장면은 무엇일까?
- 성공 케이스보다 차단/캐시/로그 케이스가 더 제품 가치를 잘 보여주지 않을까?
- "원문을 저장하지 않는데도 운영자가 추적할 수 있다"는 메시지를 어떻게 보여줄까?

### 4. 실제 Provider 연동을 할지

Mock Provider 기반은 안정적이지만, 실제 Provider 1개를 연결하면 제품 설득력이 올라갈 수 있습니다.

찬성 논리:

- "진짜 LLM Gateway"라는 느낌이 강해진다.
- 고객사 앱이 실제 LLM 응답을 받는 장면을 보여줄 수 있다.
- Mock과 실제 Provider를 같은 Gateway 흐름으로 처리한다는 메시지가 생긴다.

반대 또는 주의 논리:

- secret 관리와 장애 변수가 생긴다.
- 네트워크나 Provider 상태에 따라 데모가 흔들릴 수 있다.
- 실제 Provider 연결 자체보다 정책, 로그, 캐시, 보안 설명이 더 중요할 수 있다.

토론 질문:

- 실제 Provider는 발표 핵심 흐름에 넣을까, 보너스 시나리오로 둘까?
- 실패해도 데모가 무너지지 않도록 Mock Provider fallback을 같이 보여줄 수 있을까?
- 실제 Provider 연결을 한다면 어떤 Provider 하나가 가장 안정적이고 설명하기 쉬울까?

### 5. 고객사 앱 연동 데모

GateLM은 Chat UI 자체보다 "기존 앱의 LLM 호출 경로를 Gateway로 바꾼다"는 메시지가 강합니다.

토론하고 싶은 시나리오:

1. 고객사 앱이 직접 Provider를 호출하던 상황을 보여준다.
2. endpoint와 key를 GateLM으로 바꾼다.
3. 같은 요청이 GateLM을 통과한다.
4. 민감정보 처리, 캐시, 모델 선택, 로그가 자동으로 붙는다.
5. 운영자는 Dashboard와 Request Detail에서 결과를 확인한다.

토론 질문:

- 고객사 앱 데모를 어느 정도까지 실제처럼 만들어야 할까?
- 단순 curl / script보다 작은 Web UI가 설득력 있을까?
- 발표에서 "개발자가 바꿔야 하는 것"을 얼마나 작게 보여줄 수 있을까?

### 6. Dashboard와 Request Detail

현재 기반에서는 로그와 대시보드 조회 흐름이 이미 있으므로, 남은 시간에는 "무슨 숫자를 어떻게 보여주면 운영 제품처럼 느껴지는가"를 토론해야 합니다.

Dashboard에서 보고 싶은 것:

- 전체 요청 수
- 성공 요청 수
- 차단 요청 수
- 캐시로 절약된 요청 수
- 평균 응답 시간
- 예상 비용
- 모델별 요청 분포
- 애플리케이션별 사용량

Request Detail에서 보고 싶은 것:

- 요청 상태
- 인증된 프로젝트 / 애플리케이션 정보
- 요청 모델과 실제 선택 모델
- 선택 이유
- 캐시 상태
- 민감정보 처리 결과
- token / cost / latency 요약
- 원문이 아닌 안전한 preview 또는 summary

토론 질문:

- Dashboard는 숫자 중심이 좋을까, 데모 흐름 중심이 좋을까?
- Request Detail에서 가장 먼저 보여줘야 하는 정보는 보안일까, 비용일까, 라우팅일까?
- 운영자가 실제 장애나 차단 이유를 이해하려면 어떤 설명이 필요할까?

### 7. 보안 메시지

GateLM의 강점은 "LLM을 잘 부른다"가 아니라 "조직이 허용한 방식으로 안전하게 부른다"에 있습니다.

토론하고 싶은 기준:

- 원문 prompt / response를 저장하지 않는다는 점을 어떻게 제품 장점으로 보여줄까?
- redaction과 block을 어떤 예시로 보여줘야 과하지 않으면서 명확할까?
- 인증 실패와 권한 불일치를 운영자가 볼 수 있게 하되, 민감값은 노출하지 않는 기준은 충분한가?
- 보안 정책이 사용자를 방해하는 느낌이 아니라 보호하는 느낌으로 보이려면 어떻게 설명해야 할까?

토론 질문:

- 데모에서 일부러 위험한 값을 넣는 장면을 보여주는 것이 좋을까?
- 차단된 요청을 error처럼 보이지 않게 표현해야 하지 않을까?
- false positive는 지금 해결 기능까지 보여줄지, 운영 정책 논의로 넘길지?

### 8. 비용 절감 메시지

캐시는 이미 GateLM의 비용 절감 메시지를 보여주기 좋은 기능입니다.

토론하고 싶은 표현:

- 같은 요청을 두 번 보내고, 두 번째 요청은 Provider 호출 없이 반환된다.
- Dashboard에서 cache hit 비율이나 절약된 호출 수를 보여준다.
- Request Detail에서 왜 캐시가 적용됐는지 확인한다.
- 모델 자동 선택으로 짧은 요청은 더 저렴한 모델로 간다는 흐름을 보여준다.

토론 질문:

- 비용 절감을 실제 금액으로 보여줄까, 호출 수 감소로 보여줄까?
- cache hit가 너무 기술적으로 보이면 "절약된 요청"처럼 바꿔 말하는 게 좋을까?
- 자동 모델 선택과 캐시를 하나의 비용 절감 스토리로 묶을 수 있을까?

### 9. 테스트와 데모 안정성

남은 기간에 기능을 더 붙일 수는 있지만, 발표 흐름이 흔들리면 전체 인상이 나빠집니다.

확인하고 싶은 것:

- 정상 요청 성공
- 인증 실패 차단
- 애플리케이션 토큰 실패 차단
- 민감정보 redaction
- 위험 정보 block
- 캐시 miss 후 hit
- 자동 모델 선택
- Request Log 기록
- Request Detail 조회
- Dashboard 숫자 반영
- Mock Provider 장애 또는 timeout 상황

토론 질문:

- 발표 전에 반드시 자동으로 확인해야 하는 체크리스트는 무엇인가?
- 실제 Provider를 붙인다면 실패했을 때 어떤 대체 흐름을 준비해야 할까?
- 다른 팀원이 봐도 바로 재현 가능한 데모 script가 충분한가?

## Kyujeong의 현재 주장

- 지금은 범위를 닫을 때가 아니라, 남은 4일을 어디에 쓰면 제품 인상이 가장 커지는지 토론할 때다.
- Gateway의 핵심 흐름은 이미 어느 정도 갖춰져 있으므로, 다음 논쟁은 "더 많은 기능"보다 "더 설득력 있는 사용 장면"이어야 한다.
- 실제 Provider, Web Console, Rate Limit, Budget 중 무엇을 붙일지는 미리 배제하지 말고 impact / risk / demo value로 비교해야 한다.
- 다만 어떤 선택을 하든 보안, 캐시, 라우팅, 로그가 하나의 요청 흐름 안에서 같이 설명되어야 한다.
- v1.0.0은 기능 목록보다 "고객사 앱이 GateLM을 통과하면 무엇이 좋아지는가"를 납득시키는 쪽으로 가야 한다.

## 다른 Codex에게 던질 질문

- 너라면 남은 4일 동안 실제 Provider, Web Console, Rate Limit, Budget 중 무엇을 먼저 붙이겠는가?
- 지금 구현된 Gateway 기반에서 제품처럼 보이게 만드는 가장 빠른 방법은 무엇인가?
- 데모에서 제일 강한 장면은 보안 차단, 캐시 절감, 자동 모델 선택, 로그 추적 중 무엇이라고 보는가?
- 실제 Provider 없이도 v1.0.0 설득이 가능하다고 보는가?
- 가능하다면 어떤 시나리오로 발표 흐름을 짜겠는가?
- 불가능하다면 실제 Provider를 어떤 범위로 붙여야 한다고 보는가?
